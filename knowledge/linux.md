## Process & Thread Diagnostics

### View per-thread CPU usage

```bash
top -H -p <pid>
# or
ps -T -p <pid> -o pid,tid,pcpu,comm
```

`-H` shows threads as separate tasks. Sort by CPU with `P` inside `top`.

### View process tree

```bash
pstree -p <pid>
ps auxf
```

### Find process by port

```bash
netstat -tunlp | grep <port>
# modern alternative
ss -tunlp | grep <port>
lsof -i :<port>
```

### Kill gracefully vs force

```bash
kill -15 <pid>   # SIGTERM — graceful shutdown
kill -9 <pid>    # SIGKILL — immediate, no cleanup
```

## Memory & CPU

### System-wide memory

```bash
free -h
cat /proc/meminfo
```

Key fields: `MemTotal`, `MemAvailable`, `Buffers`, `Cached`, `SwapTotal`.

### Top memory consumers

```bash
ps aux --sort=-%mem | head -10
# sort by column 4 (memory %)
ps aux | sort -k4nr | head -3
```

### CPU core count

```bash
nproc
cat /proc/cpuinfo | grep -c processor
lscpu
```

### Load average

```bash
uptime
cat /proc/loadavg
```

Load average = runnable + uninterruptible tasks (average over 1/5/15 min). On a 8-core machine, load > 8 suggests saturation.

### Per-process memory map

```bash
pmap -x <pid>
cat /proc/<pid>/status | grep -E 'VmRSS|VmSize'
```

## I/O Multiplexing

See also [redis.md](./redis.md) for application-level usage.

### `select` / `poll` / `epoll`

| | select | poll | epoll |
|---|--------|------|-------|
| Max fds | 1024 (FD_SETSIZE) | Unlimited | Unlimited |
| fd set copy | Yes (each call) | Yes | No (kernel maintains interest set) |
| Event delivery | O(n) scan all fds | O(n) poll all | O(ready) — callback on active fds only |
| Thread-safe | No | No | Yes (epoll today) |
| Edge vs level | Level | Level | Both (edge-trigger common) |

**Why epoll is faster at scale:** After `epoll_wait`, kernel returns **only ready fds** — no linear scan of all registered connections.

**Level-triggered:** Event fires while condition holds (data in buffer).

**Edge-triggered:** Event fires on state **change** — must drain buffer completely or miss events.

Linux default for Nginx, Redis, Node.js (libuv on Linux).

## Shell & Logs

### Search logs

```bash
cat -n app.log | grep "ERROR"
grep -n "debug" test.log > debug.txt
tail -f /var/log/syslog
journalctl -u myservice -f
```

### Count occurrences

```bash
grep -c "timeout" app.log
awk '/ERROR/ { count++ } END { print count }' app.log
```

### Process substitution & pipes

```bash
diff <(sort file1) <(sort file2)
```

## Signals & Interrupts

### Hard vs Soft Interrupt

- **Hard interrupt:** Generated by hardware (NIC packet arrived, disk I/O complete). CPU jumps to ISR in kernel.
- **Soft interrupt:** Triggered by kernel or `int` instruction; handles deferred work from hard ISR (bottom halves, scheduling).

### Common Signals

| Signal | Number | Default action |
|--------|--------|----------------|
| SIGHUP | 1 | Hangup (reload config) |
| SIGINT | 2 | Interrupt (Ctrl+C) |
| SIGKILL | 9 | Kill (cannot catch) |
| SIGTERM | 15 | Terminate |
| SIGSTOP | 19 | Stop (cannot catch) |

### Kernel Mode vs User Mode

- **User mode:** Restricted instructions; memory access via virtual addresses mapped by MMU.
- **Kernel mode:** Full hardware access; runs privileged instructions, manages I/O, memory, scheduling.

Transition via **system calls**, exceptions (page fault), or hardware interrupts.
